  - "새로운 커넥터 프레임워크가 기존 커넥터 구현을 지원해야 함"

    alternatives:
      - "스레드 풀 확장 (여전히 동기 방식)"
      - "커넥터별 전용 마이크로서비스"
  - decision: "커넥터 SDK 추상화 레이어"
    reasoning: "기존 아키텍처에서는 새 커넥터마다 전송 프로토콜 처리(HTTP 재시도, OAuth 갱신, 페이지네이션)를 직접 구현해야 했습니다. SDK로 이런 공통 사항을 표준화하여 신규 커넥터 개발을 몇 주에서 며칠 수준으로 단축했습니다."
    alternatives:
      - "SDK 없이 개방형 커넥터 개발"
      - "로우코드/노코드 커넥터 빌더 UI"
  - decision: "이벤트 소싱 기반 파이프라인 실행 로그"
    reasoning: "'파이프라인이 왜 실패했나요?'라는 고객 질문에 답하기 위해, 이벤트 소싱 로그는 개별 커넥터에 계측 코드를 추가하지 않고도 모든 파이프라인 실행의 완전하고 조회 가능한 감사 추적을 제공합니다."
    alternatives:
