outcomeSummary: "Built a plant equipment monitoring system (AI PECS) that collects and normalizes 46 sensor tags from compressor/turbine equipment via a closed-network PI System, integrates externally developed AI models (diagnosis, prognosis, recommendation) into a Zeppelin-based pipeline, and visualizes results on the web in real time — validated in on-site PoCs at two client plants"
overview: "Owned the data engineering and system integration for AI PECS (AI Plant Equipment Control System), an equipment-monitoring system for petrochemical and gas plant clients (I was the PoC presenter). It collects real-time sensor data from compressor/turbine equipment via a closed-network PI System and integrates the diagnosis/prognosis/recommendation AI models — developed externally by a university research lab — into the actual operating pipeline, visualizing results on the web. Because the collected data was shared across multiple models, data consistency was the core requirement."
problem: "The AI models themselves were built by an external research team, but making them run on real equipment data required a pipeline that could collect, normalize, and reshape sensor data into model input form within a closed-network, security-isolated environment. The external (development) side only knew virtual IDs for the sensor tags — the real mapping could only be confirmed on-site — and the raw data was shared across multiple models, so consistency mattered. On top of that, anomalous data was vastly outnumbered by normal data, creating a class-imbalance problem where the model struggled to draw a clear normal/anomaly boundary."
constraints:
  - "Closed network + security isolation — the external dev side cannot know the real sensor mapping; it is confirmable only on-site"
  - "Runs on on-site hardware with low-spec PCs and limited server headroom"
  - "Raw data is shared across multiple AI models — data consistency is essential"
  - "Anomalous data is extremely scarce — class imbalance"
  - "Data-availability conditions differ per client (real data exportable or not)"
approach: "Collected raw data from the PI System by mapping the externally assigned virtual sensor IDs to the real IDs confirmed on-site, inside a Python polling script. Because the raw data is shared and consistency matters, it was landed first in the existing MSSQL, while the app's RDB was separated out as PostgreSQL. Each sensor was min-max normalized against its calibration range (min/max), and time series were turned into sliding-window sequences via librosa framing to reshape them into the AI model input shape. The diagnosis (reconstruction-error-based anomaly detection), prognosis, and recommendation models were applied in sequence, results were landed in PostgreSQL, and the web front visualized all 46 tags on the equipment diagram in real time. Batch cadence was run via Zeppelin's Cron feature."
keyDecisions:
  - decision: "Separate storage roles — land raw data in MSSQL first, use PostgreSQL as the app RDB"
    reasoning: "The raw data is shared across multiple models and consistency matters, so it was landed first in the existing MSSQL, while the RDB the app reads and updates was separated out as PostgreSQL. PostgreSQL is open source with no licensing burden — clean to adopt and maintain — and stays lightweight on low-spec on-site servers while being more than enough for the workload."
    alternatives:
      - "A single DB managing both raw and app data"
      - "Scaling out a commercial RDB"
  - decision: "Chose Apache Zeppelin as the data processing/batch tool (over Airflow)"
    reasoning: "Airflow was a candidate, but the closed-network site had only low-spec PCs available as servers, and Airflow's many components (scheduler, web server, metadata DB, workers) made it operationally heavy on that hardware. Zeppelin is already a module composing the web system, so it could handle collection, processing, and validation in a single note without adding infrastructure — the right-sized tool for this scale and these constraints. That said, once data volume grows and DAG dependencies get complex, moving to Airflow is the correct call."
    alternatives:
      - "Apache Airflow (heavy to operate on low-spec closed-network sites)"
      - "Custom Cron scripts"
  - decision: "Closed-network constraint → map virtual sensor IDs to real on-site IDs"
    reasoning: "Because of the security-isolated environment, the external dev side could not know the real sensor mapping, so the assigned virtual IDs were mapped to the real IDs confirmed on-site, inside the Python polling script, for collection."
    alternatives:
      - "Stationing dev staff on-site"
      - "Exporting the mapping table off-site"
  - decision: "Handle class imbalance — set the anomaly threshold (UCL) as residuals_max × ratio (1.5)"
    reasoning: "With abundant normal data and extremely scarce anomalies, the model could not draw a clear boundary. Using reconstruction error (autoencoder style), anomaly_score = mean(|input − prediction|), the threshold was set as UCL = residuals_max × 1.5 — raised somewhat toward the normal side to minimize misclassifying normal as anomaly (residuals_max differs per model and per tag)."
    alternatives:
      - "A fixed threshold"
      - "Normalization/thresholding based on the data's own min/max"

learnings:
  - "Building an AI model and integrating that model into a real operating pipeline so it actually runs are two different jobs — from the integration/operations angle, nailing data consistency and the normalization baseline (calibration range) was the crux"
  - "Technology choices should be made from the site's constraints (low-spec, closed network), not the ideal stack — that's why Zeppelin was chosen over Airflow, and you also have to know the boundary at which growing data means moving to Airflow"
  - "Normal/anomaly data imbalance is a fundamental limit on model performance — threshold correction like residuals_max×ratio is a symptomatic fix; the root cause is not having secured enough normal and anomalous data (left as an unresolved item within the project timeline)"
  - "Data-availability conditions differed per client, so validation methods differed too — for a client where real data could not be exported, synthetic test data was generated from sensor calibration ranges; it was enough for functional validation but limited for validating anomaly-detection performance"
