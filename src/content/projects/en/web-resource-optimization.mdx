outcomeSummary: "Reduced total web asset size from 57.9 MB to 15.3 MB, eliminating monthly cloud deployment plan overage charges"
overview: "Identified and resolved web resource bloat that was pushing the company homepage over the cloud hosting plan's storage and transfer limits, causing unexpected monthly overage charges. The optimization delivered a 74% reduction in total asset size with no visible quality degradation."
problem: "The company homepage was consistently generating cloud deployment overage fees each month. Investigation revealed the deployment bundle contained unoptimized image assets, duplicate font files loaded from multiple CDN sources, and build artifacts that should not have been included in production. Total asset size was 57.9 MB — far above what a typical marketing site requires."
constraints:
  - "Visual quality of images must be maintained (no perceptible degradation)"
  - "No changes to page design or content"
  - "Must fit within the existing cloud deployment plan tier to avoid an upgrade"
  - "Completed within 1 month to minimize ongoing cost"
approach: "Audited the full deployment bundle using Chrome DevTools and a bundle analyzer. Identified three primary contributors: oversized original images served without optimization (~40 MB), duplicate web fonts loaded via multiple CDN sources (~8 MB), and unminified CSS/JS build artifacts. Applied WebP conversion for all images, font source consolidation, and build pipeline optimization with automated size checks."
keyDecisions:
  - decision: "WebP conversion for all images"
    reasoning: "WebP provides 25–35% smaller file sizes than JPEG/PNG at equivalent visual quality. With 97%+ browser support, the fallback complexity is minimal while the size savings are immediate."
    alternatives:
      - "AVIF (smaller files but lower browser support)"
      - "Aggressive JPEG compression (visible quality loss)"
  - decision: "Self-host fonts instead of multiple CDN sources"
    reasoning: "The site was loading the same font family from 3 different CDN sources (Google Fonts, Adobe Fonts, and a local copy) due to accumulated configuration changes. Consolidating to a single self-hosted copy eliminated duplication and removed 3 external DNS lookups."
    alternatives:
      - "Revert to system fonts (breaking design change)"
      - "Keep multiple CDN sources but deduplicate at CDN level"
  - decision: "Automate optimization in the build pipeline"
    reasoning: "A manual one-time fix regresses as new content is added. Adding image optimization (sharp) and asset size limit checks to CI prevents future bloat from accumulating unnoticed."
    alternatives:
      - "One-time manual cleanup without CI enforcement"
      - "Scheduled audit instead of build-time gate"

  - "Node.js / sharp (image optimization)"
  - "WebP"
  - "Webpack / Vite (build optimization)"
  - "GitHub Actions (CI/CD)"

learnings:
  - "Deployment audits frequently reveal waste from accumulated 'temporary' configuration changes — a periodic audit process prevents gradual bloat"
  - "Image assets are almost always the dominant factor in web bundle size; start there when diagnosing deployment size issues"
  - "Fixing the build pipeline is more valuable than a one-time cleanup — optimization that can regress is not optimization"
  - "Cloud cost overages are often symptoms of missing monitoring, not just technical issues; the fix is observability as much as optimization"
