title: "PLEY — Child Behavior Research App Redesign"
role: "Backend · Android Engineer"
year: 2024
outcomeSummary: "Moved wearable token refresh — which had been failing because it was coupled to the client app — to a server-side scheduler, lifting data collection success from 0% to 90% in offline/low-power conditions and making the research experiment viable at all"
overview: "Owned the backend and Android improvements for PLEY, a mobile app for research into guiding healthy smartphone use in children. Built in collaboration with a child research team at a domestic university, the project was handed to me with no knowledge transfer after the previous engineer left for personal reasons. I started from a structural read of the codebase; on the surface it looked like maintenance, but the real issue was the system architecture itself — where a specific piece of functionality had been placed."
problem: "The experiment linked each child participant's wearable (step count) to their allowed app usage time: steps accrued usage time, and the treatment group could not see their remaining time in the UI while the control group could see it immediately via a progress bar — comparing how awareness of status affects behavior. The critical failure was that refreshing the access token required for the wearable vendor's API calls was handled on the client (the app). Because the refresh only ran while the app was active, real-world device behavior broke it: children's devices were killed in the background by low-power policies when running heavy apps like games, or sat unused with only intermittent Wi-Fi connectivity. Token refreshes were missed, and health-data retrieval and ingestion failed downstream — leaving the data collection success rate at effectively 0%."
constraints:
  - "Must preserve the ongoing study's existing data schema and token-management RDB (PostgreSQL)"
  - "Legacy codebase inherited with zero knowledge transfer — required a structural analysis first"
  - "Children's devices are an uncontrollable real-world environment: low-power policies, intermittent connectivity"
  - "Runs on on-premise backend infrastructure (no external cloud)"
  - "Delivery had to fit the next research cohort's schedule"
approach: "Decoupled token refresh from the client logic that only ran while the app was active, and moved it to a server-side scheduler that runs regardless of device state. On the on-premise backend API server, a Crontab job periodically validates and refreshes the per-user tokens stored in the DB. In parallel, I fixed the control group's progress bar — which intermittently failed to render — by separating the UI thread from the logic thread and improving GC handling on Android."
keyDecisions:
  - decision: "Move wearable token refresh from the client (app) to the server-side backend"
    reasoning: "Token data was managed in the RDB, but the actual refresh ran on the app, so it only worked while the app was active. That assumption collapsed against the real-world cases of children's devices — background kills under low-power policies and intermittent connectivity — and data collection failed. Moving refresh to an on-premise backend that runs on a Crontab schedule, independent of device state, lets the server keep tokens valid even when the device is offline."
    alternatives:
      - "Keep client-side refresh (saves device battery and server load, but sacrifices availability)"
      - "Work around background-execution policies from the client side"
  - decision: "Keep per-user token (access/refresh) management in the existing RDB (PostgreSQL)"
    reasoning: "To guarantee the consistency of each user's token state and let the server scheduler read and refresh from a single source. Only the refresh processing moved to the server while the store stayed put, so the change carried no migration cost."
    alternatives:
      - "Redesign the token store as a separate cache/store"
      - "Distribute storage per user"
  - decision: "Fix the missing control-group progress bar via Android UI/logic thread separation"
    reasoning: "The remaining-time progress bar that had to be visible for the control group failed to render intermittently. The cause was a processing conflict between the UI thread and the logic-execution thread, so I separated thread handling and improved GC so the experimental condition (the control group's awareness of status) held reliably."
    alternatives:
      - "Surface-level fixes such as simply adjusting the UI refresh interval"

  - "Android (Native)"
  - "Wearable Vendor API (step-count sync)"
  - "OAuth Token (access/refresh)"

learnings:
  - "Wherever you place a piece of functionality — client or server — decide only after concretely understanding that function's constraints and edge cases"
  - "An implicit assumption like 'only works while the app is active' looks fine until you feed in the real usage pattern (background kills, intermittent connectivity) — then the placement collapses"
  - "In legacy handed over with no knowledge transfer, look for the root cause in the architecture's functional placement, not the symptom (data loss)"
  - "Making an experimental design (treatment/control) hold requires treating client details like a UI thread bug as part of the research conditions themselves"
